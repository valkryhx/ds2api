# 工具调用规范（对齐当前运行时）

本规范分为两层：

1. `LLM 推荐输出`：稳定性最高，优先使用。  
2. `运行时兼容解析`：兜底能力，避免漏拦截，但不保证每种畸形格式都能恢复。

---

## 1. 推荐输出（唯一主路径）

当且仅当要调用工具时，整段回复应只输出一个 JSON 对象：

```json
{
  "tool_calls": [
    {
      "name": "tool_name",
      "input": {
        "param": "value"
      }
    }
  ]
}
```

硬性建议：

1. 回复只包含该 JSON，不加前后解释。
2. 顶层键必须是 `tool_calls`，值必须是数组。
3. 每个调用项使用 `{"name":"...","input":{...}}`。
4. 所有键名使用双引号。
5. 无参数时 `input` 也写 `{}`。

---

## 2. 运行时兼容解析（兜底）

当前运行时还会尝试解析以下格式：

1. 混合文本中的 JSON tool payload（含部分修复场景）。
2. `Tool use:` / `tool call:` / `function.name:` + `function.arguments:`。
3. XML/Markup 形式：`<tool_call>`、`<function_call>`、`<invoke name=\"...\">`、`<function>...</function>`。
4. 多个 `<invoke ...>...</invoke>` 连续出现的场景。

注意：

1. 兼容解析是兜底，不是首选协议。
2. 标签错配、截断、严重破损的 XML/JSON 仍可能只恢复部分调用或无法恢复。

---

## 3. 策略规则（重要）

当前行为：

1. 非 `tool_choice=none` 场景：解析到的调用默认放行，不因 allow-list 名称不匹配而直接丢弃。
2. `tool_choice=none`：结构化工具调用会被硬阻断。
3. 放行不等于一定可执行。下游执行器是否有对应工具，取决于工具名是否真实可路由。

因此：

1. 即使有兜底放行，也应尽量输出下游真实注册名（例如 `functions.shell_command`），避免执行阶段找不到工具。

---

## 4. 反例与边界

错误示例（标签错配，可能只能恢复后半段）：

```xml
<function>
  <name>shell</name>
  <parameter name="command">...</parameter>
  </invoke>
  <invoke name="shell">...</invoke>
</function>
```

建议改为结构完整、可重复的写法：

```xml
<function>
  <invoke name="functions.shell_command">
    <parameter name="command">["powershell.exe","-Command","dir"]</parameter>
    <parameter name="workdir">"D:\\git_repos\\ds2api"</parameter>
  </invoke>
</function>
```

---

## 5. 给 LLM 的最小指令

```text
When calling tools, output ONLY one raw JSON object with top-level key "tool_calls".
Do not output prose, markdown fences, XML tags, or pseudo markers.
Each call item must use {"name":"...","input":{...}}.
Use the exact downstream registered tool name whenever possible.
```
