# 工具调用规范

本规范用于约束 LLM 的工具调用输出，目标是让运行时稳定拦截并执行工具，避免“看起来像调用、实际未执行”的情况。

## 1. 唯一推荐输出格式

当且仅当你要调用工具时，整段回复必须是一个纯 JSON 对象（可以有换行和缩进），格式如下：

```json
{
  "tool_calls": [
    {
      "name": "tool_name",
      "input": {
        "param": "value"
      }
    }
  ]
}
```

硬性要求：

1. 整个回复只能是该 JSON 对象，不得有任何前后解释文字。
2. 顶层必须是 `"tool_calls"`，且值必须是数组 `[]`。
3. 每个调用项必须包含 `"name"`（工具名，字符串）。
4. 每个调用项必须包含 `"input"`（参数对象；无参数时也写 `{}`）。
5. 所有键名必须用双引号。
6. 回复必须以 `{` 开始，以 `}` 结束。

## 2. 多工具调用规范

同一轮要调用多个工具时，必须放在同一个 `tool_calls` 数组里：

```json
{
  "tool_calls": [
    { "name": "mcp__exa__web_search_exa", "input": { "query": "DeepSeek V4 测评", "num_results": 5 } },
    { "name": "Bash", "input": { "command": "date +%Y-%m-%d" } }
  ]
}
```

## 3. 严禁输出（即使兼容层偶尔能修复）

以下格式禁止生成：

1. JSON 前后加解释文字，例如：`我来调用工具...` + JSON。
2. Markdown 代码块包裹，例如 ```json ... ```。
3. XML/标签调用，例如 `<tool_call>...</tool_call>`、`<function_call>...</function_call>`。
4. 文本 KV 风格，例如 `function.name:` / `function.arguments:`。
5. 伪调用标记，例如 `[调用 Read] {...}`、`[Call Read] {...}`、`Tool use:`。
6. `[TOOL_CALL_HISTORY]` / `[TOOL_RESULT_HISTORY]` 当作新调用输出。
7. 非法 JSON（单引号键名、缺失双引号、数组漏 `[]`、多余逗号等）。

## 4. 正反例

正确：

```json
{"tool_calls":[{"name":"Read","input":{"file_path":"D:/git_codes/google_adk_helloworld_git/111.md"}}]}
```

错误（你当前遇到的问题类型）：

```text
The user wants me to ...
<tool_calls>
<tool_c name="Read">
<parameter name="file_path" string="true">D:\git_codes\google_adk_helloworld_git\111.md</parameter>
</tool_calls>
```

错误（有说明文字）：

```text
我将调用工具：
{"tool_calls":[{"name":"Read","input":{"file_path":".../111.md"}}]}
```

## 5. 工具调用后的行为

1. 收到工具结果后，应基于结果直接给出答案。
2. 仅在“结果缺失关键数据”或“工具报错”时，才继续调用下一次工具。
3. 不要重复调用已经在历史结果中成功返回的数据请求。

## 6. 生成策略（给 LLM 的最小指令）

可直接作为系统提示附加：

```text
When calling tools, output ONLY one raw JSON object with top-level key "tool_calls".
Do not output prose, markdown fences, XML tags, pseudo markers, or function.name/function.arguments text.
The response must start with "{" and end with "}".
Each call item must use {"name": "...", "input": {...}}.
```
